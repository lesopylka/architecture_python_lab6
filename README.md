## Схема базы данных

### Таблицы

В базе данных были созданы следующие таблицы:

### 1. users
Хранит основные данные учетной записи пользователя.

Поля:
- id — первичный ключ
- username — уникальное имя пользователя
- email — уникальный email
- password_hash — хэш пароля
- created_at — дата создания записи

### 2. profiles
Хранит дополнительные данные профиля пользователя.

Поля:
- user_id — первичный и внешний ключ на users(id)
- first_name
- last_name
- bio
- birth_date

### 3. posts
Хранит публикации пользователей.

Поля:
- id — первичный ключ
- author_id — внешний ключ на users(id)
- content
- created_at

### 4. chats
Хранит информацию о чатах.

Поля:
- id — первичный ключ
- title
- is_group
- created_at

### 5. chat_members
Связующая таблица между пользователями и чатами.

Поля:
- chat_id — внешний ключ на chats(id)
- user_id — внешний ключ на users(id)
- joined_at

Составной первичный ключ:
- (chat_id, user_id)

### 6. messages
Хранит сообщения пользователей в чатах.

Поля:
- id — первичный ключ
- chat_id — внешний ключ на chats(id)
- sender_id — внешний ключ на users(id)
- content
- created_at

### 7. friendships
Хранит связи между пользователями.

Поля:
- user_id — внешний ключ на users(id)
- friend_id — внешний ключ на users(id)
- status
- created_at

Составной первичный ключ:
- (user_id, friend_id)

### 8. sessions
Хранит refresh-сессии пользователей.

Поля:
- id — первичный ключ
- user_id — внешний ключ на users(id)
- refresh_token
- expires_at
- created_at

## Связи между таблицами

В базе данных реализованы следующие связи:

- profiles.user_id -> users.id
- posts.author_id -> users.id
- chat_members.chat_id -> chats.id
- chat_members.user_id -> users.id
- messages.chat_id -> chats.id
- messages.sender_id -> users.id
- friendships.user_id -> users.id
- friendships.friend_id -> users.id
- sessions.user_id -> users.id

## Использованные ограничения

В схеме базы данных были использованы ограничения, необходимые для поддержания целостности данных:

### PRIMARY KEY
Используются для уникальной идентификации записей во всех основных таблицах.

### FOREIGN KEY
Используются для связи таблиц между собой и обеспечения ссылочной целостности.

### NOT NULL
Применяются к обязательным полям, например:
- username
- email
- password_hash
- content
- created_at

### UNIQUE
Используются для полей:
- users.username
- users.email
- sessions.refresh_token

### CHECK
Используются для контроля корректности значений, например:
- запрет пустого имени
- запрет пустого содержимого поста или сообщения
- ограничение допустимых статусов дружбы:
  - pending
  - accepted
  - blocked
- запрет дружбы пользователя с самим собой

## Индексы

Для ускорения работы наиболее частых запросов были созданы индексы.

### Созданные индексы

- индексы по внешним ключам;
- индексы по полям, используемым в WHERE;
- индексы по полям, используемым в JOIN;
- составные индексы для запросов с сортировкой.

### Примеры индексов

- idx_posts_author_id
- idx_posts_author_id_created_at
- idx_chat_members_user_id
- idx_messages_chat_id
- idx_messages_chat_id_created_at
- idx_friendships_user_id_status
- idx_sessions_user_id_expires_at

### Назначение индексов

- ускорение выборки постов конкретного пользователя;
- ускорение выборки сообщений конкретного чата;
- ускорение поиска чатов пользователя;
- ускорение JOIN-запросов;
- ускорение выборки друзей по статусу;
- ускорение поиска активных пользовательских сессий.

## Партиционирование

В рамках работы было также рассмотрено возможное развитие схемы при увеличении объема данных.

Наиболее подходящей таблицей для партиционирования является таблица messages, поскольку количество сообщений со временем будет расти быстрее остальных.

**Предлагаемая стратегия**
Партиционирование по диапазону даты created_at, например по месяцам:
- messages_2025_01
- messages_2025_02
- messages_2025_03

**Зачем это нужно**
- уменьшение размера отдельных таблиц;
- ускорение выборки сообщений по временным промежуткам;
- упрощение администрирования больших объемов данных.
