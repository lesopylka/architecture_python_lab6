## Производительность, кеширование и rate limiting

### Анализ производительности

В проекте социальной сети наиболее часто выполняются операции получения пользователей, постов и статистики. Эти запросы являются hot paths, так как к ним часто обращаются клиенты.

Наиболее медленными операциями являются:
- запросы к MongoDB
- aggregation pipeline
- получение больших списков постов

Для системы важно уменьшить время ответа API и снизить нагрузку на базу данных.

  

### Стратегия кеширования

Для ускорения работы было реализовано кеширование через Redis.

Кешируются:
- `GET /users`
- `GET /posts`
- `GET /stats/posts-by-users`

Используется стратегия **Cache-Aside**:
1. API сначала проверяет Redis
2. если данных нет — выполняется запрос в MongoDB
3. результат сохраняется в Redis

TTL кеша:
- 60 секунд

Инвалидация кеша происходит при:
- создании пользователя
- обновлении пользователя
- удалении пользователя
- создании поста
- изменении лайков и комментариев

  

### Реализация кеширования

Redis подключён через Docker.

Пример кеширования:

```python
cached = redis_client.get(cache_key)
````

Если данные есть в кеше:

```python
X-Cache: HIT
```

Если запрос идёт в MongoDB:

```python
X-Cache: MISS
```

  

### Rate limiting

Rate limiting был реализован для endpoint:

```text
POST /messages
```

Используется алгоритм **Fixed Window Counter**.

Лимит:

* 10 запросов в минуту с одного IP

При превышении лимита сервер возвращает:

```text
429 Too Many Requests
```

Также добавляются заголовки:

```text
X-RateLimit-Limit
X-RateLimit-Remaining
X-RateLimit-Reset
```
## Требования к производительности

Для часто используемых GET-запросов ожидаемое время ответа должно быть меньше 200 мс при наличии данных в кеше.  
Система должна выдерживать повторные запросы к спискам пользователей и постов без постоянного обращения к MongoDB.

### Влияние на производительность

Кеширование уменьшает количество запросов к MongoDB и ускоряет ответы API.

Rate limiting защищает сервис от слишком большого количества запросов и уменьшает нагрузку на сервер.

### Измерение эффективности кеша

Эффективность кеширования можно измерять через hit rate:

hit rate = cache hits / (cache hits + cache misses)

В API для проверки кеша используется заголовок `X-Cache`.
Если значение `HIT`, данные были получены из Redis.
Если значение `MISS`, данные были получены из MongoDB и затем сохранены в кеш.


  

### Метрики

Для анализа производительности можно использовать:

* время ответа API
* количество запросов
* hit rate кеша
* нагрузку на MongoDB
* количество запросов с кодом 429

Hit rate показывает, как часто данные были получены из Redis, а не из базы данных.
