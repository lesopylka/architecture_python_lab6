# Описание стратегии кеширования и rate limiting

## Анализ производительности

В проекте социальной сети наиболее часто используются следующие endpoints:

- `GET /users`
- `GET /posts`
- `GET /messages`

Эти запросы являются hot paths, так как к ним чаще всего обращаются пользователи.

Наиболее медленные операции:
- запросы к MongoDB
- aggregation pipeline
- получение больших списков данных

Для уменьшения нагрузки на базу данных было реализовано кеширование.

 

## Стратегия кеширования

Для кеширования используется Redis.

Была выбрана стратегия **Cache-Aside**.

Принцип работы:
1. API сначала проверяет наличие данных в Redis
2. если данных нет — выполняется запрос в MongoDB
3. результат сохраняется в Redis

Кешируются:
- `GET /users`
- `GET /posts`
- `GET /stats/posts-by-users`

TTL кеша:
- 60 секунд

 

## Инвалидация кеша

Кеш очищается при изменении данных.

Инвалидация выполняется при:
- создании пользователя
- обновлении пользователя
- удалении пользователя
- создании поста
- добавлении лайков
- добавлении комментариев

Это позволяет избежать возврата устаревших данных.

 

## Rate limiting

Для endpoint:

```text
POST /messages
````

был реализован rate limiting.

Используется алгоритм:

* Fixed Window Counter

Хранилище:

* Redis

Лимит:

* 10 запросов в минуту с одного IP-адреса

При превышении лимита сервер возвращает:

```text
429 Too Many Requests
```

Также API возвращает заголовки:

* `X-RateLimit-Limit`
* `X-RateLimit-Remaining`
* `X-RateLimit-Reset`

 

## Влияние на производительность

Кеширование уменьшает количество запросов к MongoDB и ускоряет ответы API.

Rate limiting защищает систему от слишком большого количества запросов и уменьшает нагрузку на сервер.

 

## Метрики

Для анализа производительности можно использовать:

* время ответа API
* количество запросов
* hit rate кеша
* количество запросов с кодом `429`
* нагрузку на MongoDB и Redis

Hit rate показывает, насколько часто данные были получены из кеша вместо базы данных.
